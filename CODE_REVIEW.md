# Полное ревью кода `kosma-erp` 28/03/2026

## Что проверено
- Структура проекта (backend + frontend).
- Сборка фронтенда (`npm run build`).
- Сборка/тесты backend (`mvn test`).

## Критические проблемы

1. **Backend сейчас фактически не запускаемый как Spring Boot приложение**
    - Главный класс пустой: нет `@SpringBootApplication` и `main`-метода.
    - Файл `application.yml` пустой.
    - Итог: даже при исправленном `pom.xml` приложение не стартует как полноценный сервис.

2. **`pom.xml` использует неразрешимую parent-версию Spring Boot**
    - Указано `spring-boot-starter-parent:4.0.3`.
    - Команда `mvn test` завершается с ошибкой `Non-resolvable parent POM`.
    - Это блокирует backend CI/CD и любую серверную разработку.

## Существенные архитектурные и код-стайл проблемы

3. **Сильно платформозависимая Maven-конфигурация для очистки `node_modules`**
    - В `exec-maven-plugin` используется `cmd /c rmdir /s /q node_modules`.
    - Это работает только в Windows и ломает кроссплатформенную сборку.

4. **В репозитории присутствуют сборочные артефакты (`frontend/dist`) и установленный `frontend/node_modules`**
    - Это раздувает репозиторий, усложняет ревью, создает конфликтующие состояния окружения.
    - Обычно эти директории должны быть исключены через `.gitignore`.

5. **Формы содержат «заглушечные» блоки в production-коде**
    - В `PayerForm` присутствуют фрагменты `... полный набор полей ...`, `... полный код модала ...`.
    - Это затрудняет поддержку и не соответствует обещанной «полной логике» в комментариях.

6. **Смешивание imperative DOM-подхода с React-подходом**
    - В `Vedomost` много прямых обращений к DOM (`document.getElementById`, `querySelectorAll`, ручные listeners).
    - Это повышает риск ошибок, расхождения состояния UI и утечек слушателей.

7. **Проблемы доступности и UX в переключателе темы**
    - `onClick` назначен на иконку внутри `button`, а не на сам `button`.
    - Это ухудшает keyboard navigation и область клика.

8. **Избыточные/дублирующиеся импорты Bootstrap CSS**
    - Bootstrap подключается в `main.tsx`, а также локально в отдельных компонентах.
    - Увеличивается шум и риск непредсказуемых каскадных конфликтов при дальнейшем росте проекта.

9. **Слабая типизация в TypeScript и «мертвое» состояние**
    - В `InvoiceForm` используется `value: any`.
    - В `CounterpartyForm` есть `forceClose`, которое устанавливается, но не используется.
    - Это снижает предсказуемость и делает будущие рефакторинги рискованнее.

10. **Несоответствие стеков и потенциальная нестабильность зависимостей**
    - В `package.json` используются RC-версии React 19 и типы к ним.
    - Во время сборки появляются предупреждения от `react-router` про `"use client"` directives.
    - Для ERP-системы разумнее стабильный прод-стек без RC-пакетов, если нет осознанной причины.

## Пошаговый план устранения

### Этап 1. Восстановить работоспособность backend (приоритет P0)
1. Зафиксировать поддерживаемую версию Spring Boot parent (стабильную и доступную в Maven Central).
2. Привести `KosmaErpApplication` к стандартному виду (`@SpringBootApplication`, `main`).
3. Добавить минимальный smoke-тест запуска контекста (`@SpringBootTest`).
4. Проверить `mvn clean test` локально и в CI.

### Этап 2. Нормализовать сборку и репозиторий (P0/P1)
1. Удалить из Git `frontend/node_modules` и `frontend/dist`.
2. Добавить/проверить `.gitignore` для артефактов сборки.
3. Заменить Windows-only шаг очистки на кроссплатформенный подход (или убрать принудительную очистку вообще).
4. Разделить frontend/backend pipelines в CI.

### Этап 3. Очистить frontend от технического долга (P1)
1. Убрать дублирующиеся импорты Bootstrap (оставить единый глобальный вход).
2. Рефакторить `Vedomost`: убрать прямые DOM-манипуляции, перейти на React refs/state и управляемые компоненты.
3. Исправить обработчик переключателя темы (`onClick` на `button`, aria-атрибуты).
4. Вынести inline `<style>` в `index.css`/module.css.

### Этап 4. Повысить качество типизации и форм (P1/P2)
1. Убрать `any` и ввести явные интерфейсы (`InvoiceItem`, `PaymentStage`, DTO форм).
2. Удалить неиспользуемые поля/состояния (`forceClose`) и мертвый код.
3. Довести `PayerForm` до целостной реализации без placeholder-комментариев.
4. Добавить единый слой валидации (например, `zod` + react-hook-form или строго типизированный собственный).

### Этап 5. Стабилизация зависимостей и release-профиль (P2)
1. Перейти с RC-пакетов React/типов на стабильные версии (если нет блокирующих причин).
2. Зафиксировать версионную политику (минимум/максимум мажорных апдейтов).
3. Включить линтеры и quality gates (`eslint`, `tsc --noEmit`, `mvn test`) в pre-merge pipeline.

## Быстрые победы (можно сделать за 1 день)
- Починить `pom.xml` + main class Spring Boot.
- Удалить `node_modules`/`dist` из репозитория.
- Убрать `any` и `forceClose`.
- Исправить `onClick` для кнопки темы.
- Добавить минимальные smoke-тесты frontend/backend сборки в CI.
