# CODE_REVIEW_2 — что делать ПОСЛЕ завершения UI

Дата анализа: 2026-04-04  
Репозиторий: `kosma-erp`

## Что проверено

1. Структура репозитория (frontend + backend + CI).
2. Наличие и корректность backend CI-файла: `.github/workflows/backend-ci.yml`.
3. Локальный прогон `mvn -B clean test`.
4. Текущее состояние backend-кода и минимальных артефактов запуска.

## Ключевой факт по CI

Файл **`.github/workflows/backend-ci.yml` существует** и рабочий:
- поднимает Java 21,
- кэширует Maven,
- запускает `mvn -B test`.

Локально `mvn -B clean test` проходит успешно (BUILD SUCCESS), но сейчас это “зелёный” статус без реальной проверки бизнес-логики, потому что:
- backend-приложение фактически не реализовано,
- тестов нет (`No tests to run`).

## Пост-UI roadmap (подробный план)

Ниже — последовательность шагов, которые стоит делать **после** того, как интерфейс стабилизирован.

## Этап A. Скелет backend-приложения (P0)

### A1. Привести entrypoint к рабочему Spring Boot виду
- Сделать `KosmaErpApplication` полноценным:
    - `@SpringBootApplication`
    - `public static void main(String[] args)`
- Проверить старт приложения локально.

### A2. Минимальная конфигурация `application.yml`
- Добавить профиль `dev`.
- Задать базовые параметры:
    - `server.port`
    - `spring.application.name`
    - конфигурацию логирования.

### A3. Health endpoint
- Подключить Actuator и `/actuator/health`.
- Это позволит CI/CD и инфраструктуре понимать “жив” ли сервис.

---

## Этап B. Контракт между UI и backend (P0/P1)

### B1. Зафиксировать API-контракты
- На основе уже реализованного UI сформировать список endpoint’ов:
    - плательщик,
    - контрагент,
    - счёт,
    - ведомость.
- Описать DTO и ошибки API (единый формат).

### B2. Версионирование API
- Сразу ввести префикс `/api/v1`.
- Это упростит будущие изменения без поломки UI.

### B3. Документация контракта
- Добавить OpenAPI (springdoc) и генерируемую спецификацию.
- Синхронизировать поля DTO с текущими frontend-формами.

---

## Этап C. Данные и персистентность (P1)

### C1. Выбор БД и стратегия миграций
- Для dev: H2/PostgreSQL (лучше сразу PostgreSQL для реалистичности).
- Подключить Flyway или Liquibase.
- Первичная схема: таблицы под сущности UI (payer/counterparty/invoice/registry).

### C2. JPA-модель и репозитории
- Создать entity + repository для ключевых агрегатов.
- Сразу определить правила уникальности (ИНН, КПП, номер счёта и т.д.).

### C3. Транзакционные сервисы
- Вынести бизнес-операции в service-слой.
- Оставить controller “тонким”.

---

## Этап D. Реальная проверяемость в CI (P1)

### D1. Сделать `mvn clean test` осмысленным
Сейчас тестов нет, поэтому нужно:
- добавить `@SpringBootTest` smoke-тест,
- добавить хотя бы 2–3 unit/integration теста на сервисы и контроллеры,
- включить проверку JSON-контрактов (MockMvc/WebTestClient).

### D2. Обновить backend CI до `clean test`
- В workflow заменить `mvn -B test` на `mvn -B -ntp clean test`.
- При падении тестов сохранять surefire-репорты как артефакты.

### D3. Порог качества
- Запрет merge в main при красном backend CI.
- Для frontend-only PR backend workflow не должен быть обязательным.

---

## Этап E. Безопасность и базовая эксплуатация (P1/P2)

### E1. Обработка ошибок
- Ввести `@ControllerAdvice` + единый формат ошибок:
    - `code`, `message`, `details`, `timestamp`, `traceId`.

### E2. Валидация входных данных
- Использовать `jakarta.validation` в DTO.
- Валидировать данные согласованно с frontend-валидацией.

### E3. CORS/безопасность
- Настроить CORS только для frontend origin.
- Подготовить фундамент под аутентификацию (JWT/session) позже.

### E4. Логирование
- Корреляционные ID в логах.
- Разделить уровни логов для dev/prod.

---

## Этап F. Интеграция UI ↔ backend (P2)

### F1. API client во frontend
- Вынести HTTP-клиент в отдельный слой (`api/`), не смешивать с компонентами.
- Типизировать ответы backend DTO.

### F2. Поэтапная замена mock/alert-логики
- Формы сейчас демонстрационные; заменить `alert` и заглушки на реальные запросы.
- Добавить состояния loading/error/success.

### F3. Контрактные e2e сценарии
- Smoke e2e: создание плательщика, контрагента, счёта, просмотр в реестре.

---

## Этап G. Release discipline (P2)

### G1. Версионная стратегия
- Сохранить правила из `docs/dependency-version-policy.md`.
- Для backend зависимостей — обновления отдельными PR.

### G2. Definition of Done (для backend задач)
Задача считается завершённой, если:
1. Есть тесты.
2. `mvn clean test` зелёный локально и в CI.
3. Есть обновлённая документация API/архитектуры.
4. Нет breaking change для текущего UI.

---

## Рекомендуемый порядок выполнения (после UI)

1. A1 → A2 → D1 (чтобы быстро получить реальный baseline тестов).
2. B1 → B2 → B3 (чёткий контракт UI↔backend).
3. C1 → C2 → C3 (данные и сервисы).
4. E1 → E2 → E3 → E4 (стабильность и эксплуатация).
5. F1 → F2 → F3 (переход от демо UI к рабочему продукту).
6. D2 → D3 + G1/G2 (финальная дисциплина поставки).

---

## Короткий вывод

Даже при зелёном `mvn clean test` сейчас backend-проверка формальная. После завершения UI главный фокус:
1) сделать backend исполняемым,
2) закрепить API-контракт,
3) наполнить CI реальными тестами,
4) интегрировать UI с backend без ломки UX.
